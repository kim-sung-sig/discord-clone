목차

T171-B MFA 적용 범위 결정 및 구현 보고서

보안 대시보드, operator token lifecycle, 고위험 운영 동작의 step-up MFA 정책 및 구현 결과

핵심 요약

결정 및 구현 결과는 전체 `/security` 조회에 즉시 MFA를 강제하기보다, operator token 발급/폐기와 향후 고위험 보안 동작에 step-up MFA를 먼저 요구하는 것이다.

  • 현재 상태: `/security` 읽기 경로는 기존 guard를 유지하고, operator token exchange/revoke는 신선한 MFA evidence를 요구한다.
  • 위험 지점: operator token bootstrap/exchange는 short-lived token과 audit를 갖췄지만, bootstrap secret 또는 관리자 세션이 탈취되면 두 번째 확인 없이 고위험 동작으로 이어질 수 있다.
  • 구현 방향: backend auth 또는 외부 IdP가 검증한 MFA evidence를 principal claim으로 전달하고, Nuxt route policy가 freshness를 검증한다.
  • 보안 경계: production에서는 static MFA bypass를 허용하지 않고, local/QA fixture는 명시적인 non-production 설정에서만 가능하다.

I. 검토 범위

대상 화면/security dashboard, operator token form, operator token audit panel
대상 API/api/security/operator-token/exchange, /revoke, /audit, CSP telemetry/ack routes
결정 대상MFA 적용 범위, MFA 증거 출처, recovery/break-glass, local QA fixture 방식
제외 범위실제 TOTP/WebAuthn 구현, 외부 IdP 연동 구현, recovery code 저장소 구현

II. 현재 구현 관찰

영역현재 관찰MFA 결정에 미치는 영향
Dashboard access guard authorizeSecurityDashboardAccess가 backend verifier, JWT, issued operator token, static operator token, local-dev-open 순서로 평가한다. 확장 완료 MFA evidence와 freshness metadata를 principal에 추가했다.
Operator token exchange configured bootstrap token과 신선한 MFA evidence를 확인한 뒤 15분 기본 TTL의 issued operator token을 발급한다. 보호됨 bootstrap secret만으로는 issued operator token을 발급할 수 없다.
Operator token revoke/audit issued operator token store는 token hash와 audit row를 저장하고, UI는 12-character prefix만 표시한다. 유지됨 MFA gate가 추가돼도 raw token은 payload/UI에 노출하지 않는다.
Local development guard가 구성되지 않았고 production/required mode가 아니면 read-only local-dev-open이 가능하다. 분리됨 operator action MFA fixture는 non-production 명시 설정에서만 동작한다.

III. 위험 분석

R1. 관리자 세션 탈취 후 고위험 보안 동작 실행
기본 dashboard guard만으로는 최근 MFA를 거친 사용자와 단순 bearer/JWT 또는 operator token 사용자를 구분하지 못한다.
R2. Bootstrap secret 탈취 시 short-lived token 발급 가능
T129/T170 이후 발급 토큰은 짧고 감사 가능하지만, bootstrap exchange 자체에 추가 확인이 없으면 발급 진입점이 높은 가치를 가진다.
R3. 전체 dashboard MFA 강제는 사고 대응 지연 가능
읽기 전용 CSP telemetry나 guard health 확인까지 매번 MFA를 요구하면 운영자가 incident state를 빠르게 확인하기 어렵다.
R4. Frontend-only MFA는 보안 경계가 될 수 없음
브라우저 상태, sessionStorage, UI flag만으로 MFA를 판단하면 사용자가 임의 조작할 수 있다. MFA evidence는 backend 또는 IdP가 서명/검증해야 한다.

IV. 결정 옵션

옵션내용장점단점권장
A. 고위험 동작 step-up MFA operator token exchange/revoke와 향후 security action에 MFA freshness를 요구한다. 읽기 전용 dashboard는 기존 guard로 유지한다. 위험 감소와 운영 응답성을 균형 있게 맞춘다. 기존 access guard를 route policy로 확장하기 쉽다. route별 risk classification과 MFA claim 계약이 필요하다. 권장
B. 전체 `/security` MFA dashboard 진입과 모든 security API 호출에 MFA를 요구한다. 가장 단순하고 강한 보호 모델이다. incident 조회와 guard health 확인도 느려지고 break-glass UX가 복잡해진다. 강화 모드
C. MFA 유예 현재 short-lived operator token, audit, fail-closed guard만 유지한다. 구현 비용이 없다. bootstrap secret 또는 관리자 세션 탈취 시 두 번째 확인이 없다. 비권장

V. 권장 결정안

T171-B 결정: phase 1에서는 high-risk security operation에 step-up MFA를 요구한다. MFA evidence는 backend auth 또는 외부 IdP가 검증한 principal claim으로 전달하고, Nuxt는 해당 claim을 route policy에서 소비한다. production에서는 static MFA bypass를 허용하지 않는다.
결정 항목결정값구현 결과
MFA 적용 범위 operator token exchange/revoke, future security action, global role/security operator mutation exchange/revoke route가 fresh MFA evidence를 요구한다.
Read-only dashboard 기존 dashboard guard 유지. 단, production에서는 configured guard 필수 telemetry/audit read route는 기존 dashboard guard로 유지한다.
MFA 방식 backend/IdP claim 우선. TOTP/WebAuthn 자체 구현은 별도 auth backend task mfaVerified, authTime, amr/acr, auth_time claim을 정규화한다.
Recovery와 break-glass production bypass 없음. break-glass는 별도 audited principal 또는 approved emergency token만 허용 raw token, recovery code, MFA secret를 응답/문서에 추가하지 않았다.
Local/QA 명시적 non-production fixture로 MFA evidence 시뮬레이션 NUXT_SECURITY_DASHBOARD_ALLOW_TEST_MFA_HEADER는 production에서 비활성화된다.

VI. 결정 결과

  1. phase 1 MFA 적용 범위는 operator token lifecycle 같은 고위험 동작으로 확정했다.
  2. MFA evidence는 backend auth 또는 외부 IdP claim만 신뢰하고, Nuxt/browser-only state는 신뢰하지 않는다.
  3. production에서 static MFA bypass를 금지한다.
  4. local/QA에서는 명시적인 non-production fixture로 MFA evidence를 시뮬레이션할 수 있다.

VII. 구현 결과

사용자 승인 후 다음 RED/GREEN 순서로 구현했다.

  • operator token exchange/revoke가 MFA evidence 없는 principal을 거부하는 RED 테스트를 추가했다.
  • backend/JWT principal에 MFA evidence와 freshness metadata를 담는 access guard를 추가했다.
  • production에서 MFA disable/test header 우회가 동작하지 않는 설정 테스트를 추가했다.
  • UI는 MFA secret/recovery code를 저장하거나 표시하지 않고 step-up required 상태만 표시한다.
  • exchange/revoke route factory를 추가해 고위험 action policy를 직접 검증한다.
