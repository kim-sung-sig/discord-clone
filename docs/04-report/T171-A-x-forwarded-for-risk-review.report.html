T171-A X-Forwarded-For 위험성 검토 보고서

운영 proxy trust boundary, rate-limit subject, 보안 대시보드 노출 정책 결정 및 적용 기록

핵심 요약

권장 결정은 명시적 trusted proxy allowlist만 신뢰하고, 그 밖의 X-Forwarded-For는 완전히 무시하는 것이다.

  • 적용 상태: 2026-05-31 안전 기본안으로 진행 요청을 받아 Spring API rate-limit 경로도 명시적 allowlist 정책으로 정렬했다.
  • 결정 결과: 운영 환경은 trusted proxy CIDR을 명시해야 하며, allowlist 밖의 X-Forwarded-For/X-Real-IP는 무시한다.
  • 노출 정책: 보안 UI/보고서/일반 로그에는 raw IP/header를 계속 비노출로 유지한다.

I. 검토 범위

대상 헤더X-Forwarded-For, X-Real-IP, direct remote address
보호 대상CSP report limiter subject, backend API rate-limit subject, 보안 대시보드 diagnostics
제외 범위실제 proxy CIDR 값, 운영 로그 원문, raw IP/header 샘플
근거 문서T104, T128, T170, T171-A 계획 및 현재 코드 경계

II. 현재 구현 관찰

영역현재 관찰결정에 미치는 영향
Nuxt CSP limiter NUXT_CSP_RATE_LIMIT_TRUSTED_PROXY_CIDRS가 비어 있으면 forwarded header를 무시한다. exact IP와 IPv4 CIDR allowlist를 지원한다. 양호 운영 CIDR을 명시하면 spoofing 방어와 원본 client grouping을 함께 달성할 수 있다.
Nuxt diagnostics 대시보드는 source, trusted proxy match, rule count, header presence, subject hash prefix만 보여준다. 유지 권장 raw IP/header 노출 없이 운영 설정 오류를 진단할 수 있다.
Spring API rate limit DISCORD_TRUSTED_PROXY_CIDRS로 명시한 peer만 trusted proxy로 간주하고, 그 밖의 forwarded header는 무시한다. 적용됨 Nuxt CSP 경로와 동일한 명시 allowlist 모델로 정렬됐다.
문서/운영 상태 T171-A ADR은 accepted/implemented 상태이며 production startup guard와 회귀 테스트가 추가됐다. 완료 이후 변경은 같은 정책을 기준으로 검토한다.

III. 위험 분석

R1. spoofed forwarded header로 rate-limit subject가 분산될 수 있음
trusted proxy 밖의 요청에서 forwarded header를 신뢰하면 공격자가 요청마다 client IP처럼 보이는 값을 바꿔 동일 주체 제한을 회피할 수 있다.
R2. 사설망 proxy 자동 신뢰는 운영망 경계와 다를 수 있음
backend API limiter가 사설망 주소를 자동 신뢰하면, 배포 토폴로지에 따라 애플리케이션 앞단이 아닌 내부 호출자도 forwarded header 영향권에 들어갈 수 있다.
R3. raw IP/header 노출은 감사 편의보다 개인정보/운영정보 노출 비용이 큼
대시보드나 보고서에 원문 IP/header를 표시하면 보안 운영자가 빠르게 확인할 수 있지만, 저장/노출/공유 범위가 넓어져 추가 통제가 필요하다.
R4. proxy 설정 오류가 silent fallback이면 운영자가 늦게 발견함
잘못된 CIDR, 빠진 proxy peer, IPv6 CIDR 미지원 같은 설정 이슈가 배포 단계에서 잡히지 않으면 production에서 limiter grouping이 예상과 다르게 동작할 수 있다.

IV. 결정 옵션

옵션내용장점단점권장
A. 명시 allowlist 운영 proxy IP/CIDR만 설정으로 신뢰하고, 그 밖의 forwarded header는 무시한다. spoofing 방어가 명확하고 감사 가능하다. Nuxt CSP 경로의 기존 모델과 맞다. 운영 토폴로지 변경 시 설정 업데이트가 필요하다. 권장
B. 사설망 자동 신뢰 유지 loopback/site-local/unique-local peer를 proxy로 간주한다. 로컬/내부 개발 환경에서 설정 부담이 낮다. 운영망에서 신뢰 경계가 넓어질 수 있고 배포별 차이가 숨겨진다. 개발 전용
C. forwarded header 전면 비활성 모든 환경에서 direct remote address만 사용한다. spoofing 위험은 가장 낮다. load balancer 뒤에서는 모든 사용자가 동일 proxy subject로 묶일 수 있다. 임시 안전 모드

V. 권장 결정안

T171-A 기본 결정안: production에서는 explicit trusted proxy allowlist를 필수로 삼고, allowlist 밖의 X-Forwarded-For/X-Real-IP는 완전히 무시한다. 보안 UI와 보고서에는 raw IP/header를 계속 비노출로 유지한다. production proxy 설정 오류는 배포 차단 조건으로 올린다.
결정 항목권장값승인 후 구현 후보
신뢰 proxy 목록 운영 환경별 explicit CIDR allowlist backend API rate-limit도 동일한 allowlist 모델로 정렬
비신뢰 forwarded header 완전 무시 Spring 필터 회귀 테스트 추가
UI/문서 노출 raw IP/header 비노출, source/boolean/hash prefix만 허용 보안 대시보드/보고서 노출 계약 테스트 유지
배포 차단 production proxy config contract 추가 QA harness 또는 config contract에 proxy guard 추가

VI. 결정 결과

  1. production trusted proxy CIDR allowlist는 필수 정책이다.
  2. allowlist 밖에서 들어온 X-Forwarded-For와 X-Real-IP는 무시한다.
  3. 보안 UI/보고서/로그에서 raw IP와 raw forwarded header는 비노출로 유지한다.
  4. production proxy 설정 누락 또는 잘못된 CIDR은 startup guard로 차단한다.
