# T38 CSP Reporting & Style Policy Hardening Design

작성일: 2026-05-17  
PDCA Phase: Design  
Slice: T38 CSP Reporting & Style Policy Hardening

## Architecture Decision

Treat CSP reporting as security telemetry, not product analytics. Reports should be accepted through a narrow endpoint, normalized, redacted, rate-limited, and written to QA/security logs with request correlation. The browser policy stays generated by the existing Nuxt security header boundary.

## CSP Endpoint Contract

Endpoint candidates:

```text
POST /api/security/csp-report
POST /api/security/csp-report-only
```

Accepted content types:

- `application/csp-report`
- `application/reports+json`
- `application/json`

Controls:

- max body size,
- strict JSON parsing,
- no cookies/tokens persisted,
- rate limit by IP/user/session where available,
- safe response with no report echo.

## Report Normalization

Normalized fields:

| Field | Source | Notes |
| --- | --- | --- |
| `requestId` | server/generated | Correlates with logs |
| `documentUriOrigin` | `document-uri` | Origin only, path stripped by default |
| `blockedUriOrigin` | `blocked-uri` | Origin only; inline/eval kept as literals |
| `violatedDirective` | report | Required when available |
| `effectiveDirective` | report | Required when available |
| `disposition` | report | `enforce` or `report` |
| `userAgentHash` | request | Hash only |

Discard:

- full URLs with query strings,
- cookies,
- authorization headers,
- script samples by default,
- referrer query strings,
- raw request body after normalization.

## Policy Generation

Extend HTML CSP generation with optional reporting configuration:

```text
Content-Security-Policy: ...; report-to csp-endpoint; report-uri /api/security/csp-report
Content-Security-Policy-Report-Only: ...; report-to csp-report-only; report-uri /api/security/csp-report-only
Reporting-Endpoints: csp-endpoint="/api/security/csp-report", csp-report-only="/api/security/csp-report-only"
```

Report-only policy should be stricter than enforce policy when testing style hardening.

## Style Policy Hardening

Current state:

- Script inline execution is nonce-backed after T26.
- Style inline allowance remains for Nuxt/runtime CSS compatibility.

Evaluation path:

1. Inventory inline `<style>` tags and style attributes in rendered HTML.
2. Check whether framework-generated styles can receive nonce.
3. For static inline styles, evaluate hash-based CSP.
4. Keep `style-src 'unsafe-inline'` only where framework constraints require it.
5. Document every remaining exception and link it to a future removal condition.

## QA Strategy

- Unit tests for report normalization and redaction.
- Unit tests for CSP header generation with report/report-only directives.
- Abuse tests for oversized/malformed report payloads.
- Browser smoke that intentionally triggers a benign report-only violation and verifies artifact capture.
- Regression tests for T26 script nonce behavior and Nuxt hydration.

## Risks

- Browser CSP report formats differ; normalization must accept both classic and Reporting API shapes.
- Reports can contain sensitive URLs; default to origin/path stripping.
- Report endpoints can become abuse targets; rate limiting and max body size are mandatory.
- Removing style `unsafe-inline` too aggressively can break Nuxt-injected styles.
