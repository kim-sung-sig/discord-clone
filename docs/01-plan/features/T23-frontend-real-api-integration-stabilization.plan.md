# T23 Frontend Real API Integration Stabilization Plan

작성일: 2026-05-15  
PDCA Phase: Plan  
Slice: T23 Frontend Real API Integration Stabilization

## Executive Summary

| 관점 | 내용 |
| --- | --- |
| Problem | frontend real-backend smoke는 존재하지만 request correlation과 error/retry policy가 UI state에 명확히 남지 않는다. |
| Solution | shell store가 backend request id를 생성/전달/표시하고, 실패 시 accessible error에 correlation id를 포함한다. |
| Function UX Effect | 사용자는 API 실패를 role=alert로 보고, 운영자는 같은 request id로 backend logs를 추적할 수 있다. |
| Core Value | UI success는 backend 성공 후에만 반영되고, 실패는 재시도 가능한 상태와 상관관계 id를 남긴다. |

## Scope

- REST-backed shell actions keep mutating state only after backend success.
- Browser access token remains Pinia memory only.
- Request ids are generated by frontend store and sent as `X-Request-Id`.
- API errors include accessible message and request id.
- Mocked unit tests and real-backend Playwright tests remain separated.

## Out of Scope

- Full replacement of every Pinia-only action.
- WebSocket gateway transport.
- Token refresh persistence.

## Success Criteria

- login -> create guild/channel -> send message -> voice/stage smoke remains REST-backed.
- frontend request ids are visible in request headers and error state.
- API errors are surfaced accessibly in UI.
- mocked and real-backend tests are clearly separated.

## Failure Criteria

- UI can show success while backend rejected the action.
- access tokens are persisted insecurely.
- Playwright only validates local store mutations.
